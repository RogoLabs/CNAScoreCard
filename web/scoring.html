EAS Scoring Methodology

The Enhanced Aggregate Scoring (EAS) evaluates CVE record quality across six key dimensions, with a maximum score of 100 points. Each CVE is published by a CNA (CVE Numbering Authority).

1. Foundational Completeness (30 points)

Measures the presence of basic, essential information needed to understand and act on a vulnerability.

Metric Points Description
Description Quality 15 Advanced content analysis evaluating technical depth, specificity, and clarity
Affected Products 10 Clear identification of affected products
Version Information 5 Specific version ranges or status information
Example: A CVE that specifies "Apache HTTP Server versions 2.4.0 through 2.4.52" with a detailed description like "A buffer overflow vulnerability in the mod_rewrite module allows remote attackers to execute arbitrary code via crafted HTTP requests when processing malformed URL patterns" would score the full 30 points.

2. Root Cause Analysis (10 points)

Evaluates the technical depth of vulnerability classification and problem type identification.

Metric Points Description
CWE Classification 5 Valid CWE (Common Weakness Enumeration) identifier
Problem Type Detail 5 Additional technical context or detailed problem classification
Example: A CVE with CWE-120 (Buffer Copy without Checking Size of Input) provides clear categorization for developers to understand the vulnerability pattern.

Description Quality Evaluation

The system uses advanced content analysis to evaluate description quality across multiple dimensions (15 points total):

Quality Indicator Points Examples
Basic Length & Structure 3 Progressive scoring for descriptions ≥50, ≥100, ≥200 characters
Technical Vulnerability Type 4 "buffer overflow", "SQL injection", "XSS", "privilege escalation", "code injection"
Impact/Exploitation Context 4 "allows", "enables", "leads to", "can be exploited", "unauthorized access", "execute"
Technical Specificity 4 "function", "parameter", "API endpoint", "when processing", "via the", "module"
Generic Content Penalty -2 Multiple generic phrases in short descriptions are penalized
High Quality (12-15 points): "A buffer overflow vulnerability in the mod_rewrite module of Apache HTTP Server allows remote attackers to execute arbitrary code via crafted HTTP requests when processing malformed URL patterns in .htaccess files."

Medium Quality (6-11 points): "A SQL injection vulnerability in the user login function allows authenticated users to execute arbitrary SQL commands."

Low Quality (1-5 points): "A security vulnerability exists in the application that may allow unauthorized access."

2. Root Cause Analysis (10 points)

Evaluates whether the CVE provides insight into the underlying weakness type.

Metric Points Description
CWE ID Provided 10 Valid CWE identifier (e.g., CWE-79, CWE-120)
Example: A CVE that includes "CWE-787: Out-of-bounds Write" provides developers with the specific weakness pattern to look for.

3. Software Identification: (10 points)

Assesses whether the CVE record includes a valid CPE identifier for affected products, enabling precise software identification and automation.

Metric Points Description
CPE Present 10 Valid CPE identifier (e.g., cpe:2.3:a:apache:http_server:2.4.52:*)
Example: Including "cpe:2.3:a:apache:http_server:2.4.52:*:*:*:*:*:*:*" enables automated vulnerability scanning tools to identify affected systems.

4. Severity & Impact Context (25 points)

Assesses the quality and completeness of severity scoring information.

Metric Points Description
CVSS Base Score 15 CVSS v4.0/v3.1/v3.0 base score provided
CVSS Vector String 5 Complete CVSS vector string for reproducibility
Impact Description 5 Description includes impact indicators
Example: A CVE with CVSS v4.0 or v3.1 base score of 9.8, complete vector string, and description mentioning "remote code execution" would score 25 points.

5. Actionable Intelligence (20 points)

Measures the availability of information that enables immediate action by security teams.

Metric Points Description
Solution Information 8 Available fixes, patches, or mitigations
Actionable References 6 Links to patches, advisories, or security guidance
Workarounds 2 Temporary mitigation steps
Detailed Solution 4 Solution or fix description is detailed (>100 characters)
Example: A CVE with vendor advisory, security researcher blog post, patch commit, and a detailed solution description would score the full 20 points. No points are given for published exploits.

6. Data Format & Precision (5 points)

Evaluates the use of structured, machine-readable formats. Points are awarded only if all present fields are correctly formatted.
Missing fields do not penalize the score, but if a field is present and incorrectly formatted, the score is 0.

Metric Points Description
All Format Checks Pass 5 CPE identifiers, proper CVSS format with vector string, valid CWE format, language tags, and structured affected products (if present)

Format Requirements (All Present Must Pass for Points):

  • CPE Usage: If present, must be a valid Common Platform Enumeration identifier (e.g., cpe:2.3:a:apache:http_server:2.4.52:*)
    Validation performed using the python-cpe library.
  • CVSS Format: If present, must be a complete CVSS score with both baseScore and vectorString
    Validation performed using the python-cvss library (supports v2, v3, v4).
  • CWE Format: If present, must be a valid CWE identifier (e.g., CWE-120, CWE-787)
    Validation performed using a prebuilt list from MITRE CWE and checked with a local validator.
  • Language Tags: If descriptions are present, must include proper language identification (e.g., "lang": "en")
  • Structured Data: If affected products are present, must be well-formed with vendor and product fields

External Libraries Used for Verification:

  • python-cvss — for CVSS vector and score validation
  • python-cpe — for CPE identifier validation
  • MITRE CWE Catalog — for CWE ID validation

Important Notes:

  • Data Format & Precision uses all-or-nothing scoring: A CVE must meet all format requirements for present fields to receive any points in this category
  • Missing fields do not penalize the score, but present and incorrectly formatted fields result in 0 points.
  • Scores are calculated for the last 6 months: Only CVEs published in the most recent 6-month period are included
  • Machine-readable format emphasis: Higher scores favor CVEs with structured, automation-friendly data
Example: A CVE record that includes "cpe:2.3:a:apache:http_server:2.4.52:*:*:*:*:*:*:*", "CWE-787", and "CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:H" with proper language tags and structured data would receive the full 5 points. If any present field is incorrectly formatted, the score is 0. Missing fields do not penalize.

Aggregation and Ranking

CNA scores are calculated by:

  1. Individual CVE Scoring: Each CVE receives a score from 0-100 based on the metrics above
  2. CNA Average: The arithmetic mean of all CVEs published by a CNA in the last 6 months
  3. Minimum Threshold: CNAs must have published at least 1 CVE to receive a score
  4. Inactive CNAs: CNAs with no recent publications are marked as "No CVEs published in the last 6 months"
Data Freshness: Scores are updated every 6 hours using the latest CVE data from the official CVEProject repository.
Note: This project and scoring methodology were inspired by the CNA Enrichment Recognition program.

Ranking

Ranking shows a CNA's position among all active CNAs based on their average EAS score. For example, "Rank: 12 of 150" means this CNA is 12th out of 150 active CNAs in the last 6 months.

Why This Matters

Higher EAS scores indicate CVE records that are:

  • More Actionable: Security teams can quickly understand and respond to threats
  • Better Integrated: Automated tools can process structured data effectively
  • More Complete: All necessary information is provided upfront
  • Industry Leading: Following best practices for vulnerability disclosure
